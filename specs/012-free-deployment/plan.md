# Implementation Plan: Free Public Deployment

**Branch**: `012-free-deployment` | **Date**: 2026-04-03 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/012-free-deployment/spec.md`

## Summary

Deploy the PDF Chatbox application to public free-tier hosting so it is accessible to anyone without local setup. The frontend deploys to Vercel (Next.js native support, Hobby plan — free) and the backend deploys to Render (Python 3.11 Web Service, free tier). The only code change required is making the backend CORS origin list configurable via an environment variable, so the Vercel production domain can be added without a code commit. All secrets are managed through each platform's encrypted environment variable system.

## Technical Context

**Language/Version**: Python 3.11 (backend) · TypeScript (frontend)
**Primary Dependencies**: FastAPI + uvicorn (backend) · Next.js 16.2.0 App Router (frontend)
**Storage**: Supabase PostgreSQL + Supabase Storage (no change — existing)
**Testing**: Manual end-to-end verification (no new automated tests; deployment is config-only)

**Project Type**: web-service
**Performance Goals**: Handle <100 concurrent users with standard web latency; cold-start of ≤35 s acceptable on Render free tier
**Constraints**: $0/month total cost · no credit card required on any platform · Render free tier sleeps after 15 min inactivity
**Scale/Scope**: <100 monthly active users

## Implementation Phases

### Phase 1 — Code Change (CORS)

**File**: `backend/main.py`
**Change**: Replace the hardcoded `allow_origins` list with a value read from the `ALLOWED_ORIGINS` environment variable. Falls back to `http://localhost:3000` when the variable is not set, so local development is unaffected.

```python
import os

origins = [o.strip() for o in os.getenv("ALLOWED_ORIGINS", "http://localhost:3000").split(",")]

app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,
    ...
)
```

**Verification**: Run the backend locally with `ALLOWED_ORIGINS=http://localhost:3000` and confirm existing behaviour is unchanged.

---

### Phase 2 — Render Deployment (Backend)

1. Create a new **Web Service** on Render pointing to the GitHub repo
2. Set root directory to `backend/`
3. Set build command: `pip install -r requirements.txt`
4. Set start command: `uvicorn main:app --host 0.0.0.0 --port $PORT`
5. Set environment variables (see [environment-contracts.md](contracts/environment-contracts.md)):
   - `SUPABASE_URL`
   - `SUPABASE_KEY` (service_role key)
   - `OPENAI_API_KEY`
   - `ALLOWED_ORIGINS` (placeholder until Vercel URL is known)
   - `PYTHON_VERSION=3.11`
6. Deploy and note the service URL: `https://<service>.onrender.com`
7. Verify health endpoint: `GET https://<service>.onrender.com/health` → `{"status":"ok"}`

---

### Phase 3 — Vercel Deployment (Frontend)

1. Import the GitHub repo into a new Vercel project
2. Set root directory to `frontend/`
3. Set environment variables:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - `NEXT_PUBLIC_API_URL=https://<service>.onrender.com`
4. Deploy and note the project URL: `https://<project>.vercel.app`

---

### Phase 4 — Cross-Service Wiring

1. Update `ALLOWED_ORIGINS` on Render to the Vercel URL → trigger redeploy
2. In Supabase dashboard → Authentication → URL Configuration:
   - Set **Site URL** to `https://<project>.vercel.app`
   - Add `https://<project>.vercel.app/**` to **Redirect URLs**

---

### Phase 5 — End-to-End Verification

Run through the verification checklist in [quickstart.md](quickstart.md) Step 5:
1. Homepage loads at Vercel URL
2. Guest PDF upload succeeds (backend wakes up within 35 s)
3. AI chat returns a streamed response
4. User registration and login work (Supabase Auth redirect URLs configured)
5. Authenticated user PDF upload and viewing work

**Done** when all five checks pass.
